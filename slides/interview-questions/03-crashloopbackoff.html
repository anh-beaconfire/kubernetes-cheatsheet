CrashLoopBackOff

  • CrashLoopBackOff means the container starts, crashes quickly, and Kubernetes keeps restarting it with an increasing backoff delay.
  • First, confirm state and reason: check pod status and container state (Waiting/Terminated, exit code, reason).
  • Use logs to find the root cause: application exceptions, startup failures, missing configuration, or dependency errors.
  • Common causes: missing or wrong env vars, wrong command/args/entrypoint, bad image version, or misconfigured config/secret/volume.
  • Check probes: overly aggressive liveness/readiness/startup probes can kill containers before they are ready.
  • Check resources: OOMKilled or resource limits too low can cause repeated crashes.
  • Check permissions: filesystem permissions, runAsNonRoot, dropped capabilities, or missing service account permissions.
  • If needed, override command to keep the container alive (e.g., sleep) and debug inside the container.
  • Fix by updating config (env, secrets, volumes), adjusting probes/resources, or fixing the application code and image.
# 1. Confirm CrashLoopBackOff and container state
kubectl get pod <pod> -n <ns>
kubectl describe pod <pod> -n <ns>

# Look for: state=Waiting/Terminated, Reason, ExitCode, OOMKilled, probe failures

# 2. Inspect logs (current and previous restart)
kubectl logs <pod> -n <ns>
kubectl logs <pod> -n <ns> --previous

# Look for: stack traces, missing env vars, connection failures, bad config

# 3. Verify config, env, and dependencies
kubectl describe pod <pod> -n <ns> | grep -A5 "Environment"
kubectl describe pod <pod> -n <ns> | grep -A5 "Mounts"
kubectl get cm,secret,pvc -n <ns>

# 4. Check probes and resources
kubectl describe pod <pod> -n <ns> | grep -A5 -E "Liveness|Readiness|Startup"
kubectl describe pod <pod> -n <ns> | grep -i "OOMKilled"

# 5. Debug by keeping the container alive (override command)
kubectl run debug-shell \
  --image=<image> \
  -n <ns> \
  --restart=Never \
  --command -- sh -c "sleep 3600"

# Or temporarily patch the pod/deployment command to `sleep` and exec in:
kubectl exec -it <pod> -n <ns> -- sh

# After identifying the issue:
# - fix env/config/secrets/volumes
# - adjust probes/resources
# - build and deploy a corrected image
