How do you debug a failing pod?

  • Identify the pod and check its status/phase (Pending, CrashLoopBackOff, ImagePullBackOff, etc.).
  • Describe the pod to inspect container statuses(Waiting/Terminated), exit codes, probe failures, image pull errors, mount failures, missing secrets/configs.
  • Check current and previous container logs to see application-level errors: crash traces, startup failures, missing env vars, connection errors
  • Inspect image configuration: image name/tag, registry access, imagePullSecrets, and pull errors.
  • Verify configuration: environment variables, ConfigMaps, Secrets, and mounted volumes/PVCs.
  • Review readiness/liveness/startup probes for misconfigurations causing restarts or failing readiness.
  • Check node conditions and resource pressure (CPU/memory/disk) that might evict or block the pod.
  • If the pod is running but misbehaving, exec into the container or port-forward to reproduce and inspect.
# 1. Check pod status and basic info
kubectl get pod  -n  -o wide

# 2. Describe the pod for detailed state + events
kubectl describe pod  -n 

# 3. Check logs (current and previous)
kubectl logs  -n 
kubectl logs  -n  --previous

# 4. Verify image configuration
kubectl get pod  -n  -o jsonpath='{.spec.containers[*].image}'

# 5. Inspect ConfigMaps, Secrets, and PVCs
kubectl get cm, secret, pvc -n 

# 6. Check mounts inside the pod spec
kubectl describe pod  -n  | grep -A5 "Mounts"

# 7. Inspect readiness/liveness/startup probes
kubectl describe pod  -n  | grep -A5 -E "Liveness|Readiness|Startup"

# 8. Check node health and resource pressure
kubectl describe node 

# 9. Exec into the pod (if running)
kubectl exec -it  -n  -- sh
# Check: DNS resolution, network connectivity, missing files, permissions issues

# 10. Port-forward to reproduce issues locally
kubectl port-forward pod/ 8080:8080 -n 
# Use to: hit health endpoints, reproduce startup issues, inspect app behavior
