Service Not Working

  • Check selector alignment: Service selectors must match Pod labels exactly; mismatches produce zero endpoints.
  • Verify Pod readiness: only Ready Pods receive traffic; failing readiness probes cause empty endpoints.
  • Confirm correct targetPort: Service targetPort must match the containerPort the app actually listens on.
  • Check NetworkPolicies: restrictive ingress/egress rules may block traffic between Service and Pods.
  • Validate DNS resolution: ensure the Service name resolves and the namespace is correct.
  • Check application binding: apps listening on 127.0.0.1 instead of 0.0.0.0 will not receive Service traffic.
  • Inspect Ingress/Gateway: incorrect host/path rules or backend references break routing before the Service.
# 1. Check the Service definition
kubectl get svc -n <ns>
kubectl describe svc <service> -n <ns>
# Look for: selector, ports, targetPort, type

# 2. Check if the Service has endpoints
kubectl get endpoints <service> -n <ns>
# Look for: empty subsets = selector mismatch or Pods not Ready

# 3. Verify Pod labels match the Service selector
kubectl get pods -n <ns> --show-labels
# Look for: exact key/value match

# 4. Check Pod readiness and container ports
kubectl describe pod <pod> -n <ns>
# Look for: readiness probe failures, wrong containerPort

# 5. Validate DNS resolution inside the cluster
kubectl exec -it <pod> -n <ns> -- nslookup <service>
kubectl exec -it <pod> -n <ns> -- curl http://<service>:<port>
# Look for: NXDOMAIN, connection refused, timeouts

# 6. Inspect NetworkPolicies
kubectl get networkpolicy -n <ns>
kubectl describe networkpolicy <policy> -n <ns>
# Look for: missing allow rules for namespace/pod/port

# 7. Check Ingress/Gateway routing (if applicable)
kubectl describe ingress -n <ns>
kubectl get gateway,httproute -n <ns>
# Look for: wrong host/path, wrong backend service/port

# 8. Ensure the app listens on 0.0.0.0, not localhost
kubectl logs <pod> -n <ns>
# Look for: "listening on 127.0.0.1" or similar
