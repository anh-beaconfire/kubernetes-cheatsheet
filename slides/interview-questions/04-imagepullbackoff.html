ImagePullBackOff

  • ImagePullBackOff means Kubernetes tried to pull the container image but failed repeatedly.
  • Verify the image reference: wrong registry, repository, name, or tag are the most common causes.
  • Check if the tag exists: using a non-existent or mutable tag often leads to pull failures.
  • Confirm registry authentication: private registries require valid imagePullSecrets.
  • Check registry availability: network issues, DNS failures, or rate limits can block pulls.
  • Inspect imagePullPolicy: `Always` forces pulls and may fail if the registry is unreachable.
  • Fix by correcting the image name/tag, configuring imagePullSecrets, or restoring registry access.
# 1. Inspect pod events to see the exact pull error
kubectl describe pod <pod> -n <ns>
# Look for: ErrImagePull, ImagePullBackOff, unauthorized, not found, TLS errors

# 2. Verify the image reference in the pod spec
kubectl get pod <pod> -n <ns> -o jsonpath='{.spec.containers[*].image}'
# Look for: typos, wrong registry, wrong tag

# 3. Check if the imagePullSecret exists
kubectl get secret -n <ns>
# Look for: missing or incorrectly named secret

# 4. Create or update a docker-registry secret (for private registries)
kubectl create secret docker-registry regcred \
  --docker-server=<registry> \
  --docker-username=<user> \
  --docker-password=<password> \
  --docker-email=<email> \
  -n <ns>

# 5. Patch the service account or pod to use the secret
kubectl patch serviceaccount default \
  -p '{"imagePullSecrets": [{"name": "regcred"}]}' -n <ns>

# 6. If registry is unreachable, test connectivity from a node
kubectl debug node/<node> --image=busybox -- chroot /host sh
nslookup <registry>
wget https://<registry>
# Look for: DNS failures, network blocks, TLS issues
