Reliability

Liveness, Readiness, Startup Probes

  • Liveness probe: answers “Should this container be restarted?” — detects deadlocks and hard failures.
  • Readiness probe: answers “Should this Pod receive traffic?” — controls inclusion in Service endpoints.
  • Startup probe: answers “Has a slow‑starting app finished booting?” — delays liveness/readiness until startup is done.
  • Probes are defined in the container spec inside the Pod template (Deployment, StatefulSet, DaemonSet, Job, or Pod).
  • Do not make liveness too aggressive — flapping liveness can cause restart storms and hide real issues.
  • Readiness should check what’s needed to serve traffic (DB, cache, migrations, warm caches), not just “process is up”.
  • Startup probe protects slow apps by giving them time to initialize before liveness/readiness start firing.
# Probes defined inside the container of a Deployment Pod template
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  replicas: 3
  template:
    spec:
      containers:
        - name: api
          image: my-api:1.0

          # 1) Liveness: restart container if this fails repeatedly
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
            failureThreshold: 3

          # 2) Readiness: only send traffic when this passes
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            periodSeconds: 5
            failureThreshold: 3

          # 3) Startup: give slow apps time to boot
          startupProbe:
            httpGet:
              path: /startup
              port: 8080
            failureThreshold: 30   # 30 * 10s = up to 300s to start
            periodSeconds: 10
