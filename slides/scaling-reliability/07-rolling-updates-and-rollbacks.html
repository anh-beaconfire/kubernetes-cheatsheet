Reliability

Rolling Updates and Rollbacks

  • Rolling update gradually replaces old Pods with new ones, keeping the service available during deploys.
  • Use readiness probes so new Pods only receive traffic when they are truly ready.
  • Tune maxSurge and maxUnavailable to control how many extra Pods are created and how many can be down during rollout.
  • Monitor during rollout (errors, latency, saturation) and be ready to pause or roll back quickly.
  • Use immutable image tags (e.g., versioned or digest) to ensure rollbacks are deterministic.
  • Keep a rollback path available: previous ReplicaSet, image tag, or GitOps revision.
  • Align rolling strategy with PDBs and HPA so autoscaling and disruption budgets don't contradict
    • Set HPA minReplicas ≥ PDB minAvailable to avoid scale‑down blocking rollouts.
    • Use HPA scale-down stabilization to prevent autoscaling during a rollout.
    • Avoid overly strict PDBs that block node drains or stall rolling updates. 
# Deployment with rolling update strategy
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # allow 1 extra Pod above desired replicas
      maxUnavailable: 1  # at most 1 Pod unavailable during update
  template:
    spec:
      containers:
        - name: api
          image: myorg/api:1.1.0 
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080

---

# Rolling out and rolling back from the CLI
kubectl set image deployment/api api=myorg/api:1.1.0
kubectl rollout status deployment/api
kubectl rollout history deployment/api
# rollback to previous revision
kubectl rollout undo deployment/api
