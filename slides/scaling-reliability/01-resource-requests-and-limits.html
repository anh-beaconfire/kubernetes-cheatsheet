Scaling

Resource Requests and Limits

  • Requests define the minimum CPU/memory a Pod needs — the scheduler uses these values to decide where the Pod can run.
  • Limits define the maximum CPU/memory a Pod is allowed to consume — preventing noisy‑neighbor issues.
  • We are scaling per‑Pod resources: how much CPU and RAM each Pod gets, not how many Pods exist.
  • Missing requests cause poor scheduling — Pods may land on nodes without enough capacity and get evicted.
  • Wrong limits cause performance issues:
    • CPU limit too low → throttling
    • Memory limit too low → OOMKilled
    • No memory limit → Pod can crash the node
  • Always set requests — they ensure predictable scheduling and stable autoscaling behavior.
  • Set memory limits carefully — memory is not compressible; exceeding the limit kills the container.
  • Use real metrics (Prometheus, Datadog, CloudWatch) to size requests/limits based on actual usage, not guesses.
# Example: CPU and memory sizing per Pod
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  replicas: 3
  template:
    spec:
      containers:
        - name: api
          image: my-api:1.0
          resources:
            requests:
              cpu: "250m"      # scheduler guarantees 0.25 CPU
              memory: "256Mi"  # scheduler guarantees 256 MiB RAM
            limits:
              cpu: "500m"      # Pod cannot exceed 0.5 CPU
              memory: "512Mi"  # Pod is killed if it exceeds 512 MiB
