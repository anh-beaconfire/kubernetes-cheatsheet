Scaling

Cluster Autoscaler vs HPA vs VPA

Characteristic HPA VPA Cluster Autoscaler
What it scales Number of Pod replicas Pod CPU/Memory requests & limits Number of worker nodes in the cluster
Primary goal Match workload demand with Pod count Right‑size Pods for efficiency & stability Ensure enough capacity to schedule Pods
What it acts on Deployment / StatefulSet / ReplicaSet Pod template (via recommendations or auto‑apply) Node groups / autoscaling groups
Key dependency Accurate CPU/memory requests Metrics history (Prometheus / Metrics API) Pending Pods that cannot be scheduled
Metrics used CPU, memory, custom/external metrics Resource usage over time Node utilization & unschedulable Pods
Scaling direction Horizontal (more/fewer Pods) Vertical (bigger/smaller Pods) Cluster-level (more/fewer nodes)
When it triggers Pods exceed target utilization Pod usage diverges from requests Pods are Pending due to insufficient capacity
Risks / pitfalls Wrong requests → wrong scaling; flapping Pod restarts when applying new resources Slow scale-up; cost spikes; PDB blocking scale-down
Best for Traffic-driven workloads (APIs, web apps) Batch jobs, ML workloads, memory-heavy apps Any cluster with variable load
