Scaling

PodDisruptionBudget (PDB)

  • PDB protects availability by limiting how many Pods of a workload can be voluntarily disrupted (drain, upgrade, node scale‑down).
  • Use PDB for production services to avoid accidental downtime during maintenance or autoscaler actions.
  • Ensure the workload has enough replicas to satisfy the PDB — otherwise Pods become “un-evictable”.
  • Do not set impossible budgets (e.g., minAvailable: 3 when replicas = 2).
  • Combine with readiness probes so only healthy Pods count toward availability.
  • PDB only affects voluntary disruptions — not crashes, OOMKills, or node failures.
  • Cluster Autoscaler scale‑down respects PDBs — overly strict PDBs can block node removal.
# Example: ensure at least 2 Pods remain available
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: api

---

# Alternative: allow only 1 Pod to be disrupted at a time
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-pdb
spec:
  maxUnavailable: 1
  selector:
    matchLabels:
      app: api
