Troubleshooting

Metrics

  • Metrics reveal system health and performance — essential for debugging, scaling, and alerting.
  • metrics-server (used by kubectl top) provides basic CPU/memory usage for Pods and nodes.
  • Prometheus collects detailed application, cluster, and infrastructure metrics.
  • Grafana visualizes dashboards for latency, throughput, saturation, and errors.
  • Monitor CPU, memory, restarts, latency, error rate, throughput, saturation.
  • Check node pressure (disk, memory, PID pressure) to detect cluster‑level issues.
  • Alert on unavailable replicas, high latency, OOMKills, and CrashLoopBackOff.
  • Use metrics to validate autoscaling behavior (HPA/VPA) and resource requests. 

kubectl top nodes
# → Check CPU/memory pressure, node saturation

---

kubectl top pods -A
# → Identify Pods consuming excessive resources

---

# Describe Pod to see restarts, OOMKills, throttling
kubectl describe pod api-123
# → Look for: OOMKilled, RestartCount, CPU throttling

---

# Check HPA metrics and scaling decisions
kubectl describe hpa api
# → See current CPU/memory utilization and scale events

---

# Check node conditions (memory/disk/PID pressure)
kubectl describe node worker-1
# → Look for: MemoryPressure, DiskPressure, PIDPressure

---

# Query Prometheus directly (if port-forwarded)
curl http://localhost:9090/api/v1/query?query=up
# → Verify targets are scraping correctly
