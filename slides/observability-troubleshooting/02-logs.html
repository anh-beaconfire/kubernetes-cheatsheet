Troubleshooting

Logs

  • Logs show what your application is doing — they are the first place to look when debugging failures.
  • Write logs to stdout/stderr so Kubernetes can collect them automatically.
  • Use structured JSON logs for easier searching, filtering, and correlation.
  • Centralize logs (ELK, Loki, Cloud logging) to avoid losing them when Pods restart.
  • Include correlation IDs to trace a request across services.
  • Retain logs outside Pods — container logs are ephemeral and disappear on restart.
  • Check logs for errors, stack traces, timeouts, restarts, readiness failures, and latency spikes.
# Logs from a specific Pod
kubectl logs pod/api-123
# → Check for errors, crashes, startup failures

---

# Logs from all Pods in a Deployment
kubectl logs deployment/api
# → Useful when replicas share the same behavior

---

# Logs from a specific container in a Pod
kubectl logs pod/api-123 -c api
# → For multi-container Pods (sidecars, init containers)

---

# Logs from the previous container instance
kubectl logs pod/api-123 --previous
# → Critical for debugging crash loops (CrashLoopBackOff)

---

# Stream logs in real time
kubectl logs -f deployment/api
# → Watch live behavior during deploys or incidents
