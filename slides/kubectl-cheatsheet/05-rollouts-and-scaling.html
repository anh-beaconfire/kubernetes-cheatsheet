Rollouts and Scaling

  • Rollouts control how new versions of your application are deployed and how Pods are replaced.
  • Rollout status and history help verify if a deployment succeeded or if it failed partway through.
  • Undo lets you revert to a previous ReplicaSet when a deployment breaks.
  • Manual scaling adjusts replicas immediately; autoscaling reacts to CPU or custom metrics.
  • Tip: rollouts, scaling, and HPA all interact — understand how they affect each other during incidents.
# Check rollout progress (waits until complete)
kubectl rollout status deployment/api

# View previous ReplicaSets and revisions
kubectl rollout history deployment/api

# Roll back to the previous revision
kubectl rollout undo deployment/api

# Manually scale the Deployment
kubectl scale deployment/api --replicas=5

# Create an HPA (CPU-based autoscaling)
kubectl autoscale deployment api --cpu-percent=70 --min=3 --max=10
