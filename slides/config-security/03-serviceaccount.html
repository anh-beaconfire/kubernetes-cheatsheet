Configuration

ServiceAccount

  • A ServiceAccount provides a Pod with an identity for accessing the Kubernetes API or cloud‑integrated permissions (IAM roles, workload identity).
  • Avoid using the default ServiceAccount — it often has broader permissions than needed and is shared across Pods.
  • Disable automountServiceAccountToken when Pods do not need API access to reduce attack surface.
  • Follow least privilege — bind only the minimal RBAC permissions required for the workload.
  • Use Role and RoleBinding for namespace‑scoped permissions; use ClusterRole only when absolutely necessary.
  • For cloud environments, map ServiceAccounts to cloud IAM roles (GCP Workload Identity, AWS IRSA, Azure Workload Identity) instead of embedding long‑lived credentials.
  • Rotate tokens and audit ServiceAccount usage regularly to detect unexpected API calls.
# ServiceAccount definition
apiVersion: v1
kind: ServiceAccount
metadata:
  name: api-service-account
automountServiceAccountToken: false   # disable token unless needed

---

# Pod using the ServiceAccount
spec:
  serviceAccountName: api-service-account

---

# Example: RBAC binding (least privilege)
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: api-reader
rules:
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "list"]

---
kind: RoleBinding
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: api-reader-binding
subjects:
  - kind: ServiceAccount
    name: api-service-account
roleRef:
  kind: Role
  name: api-reader
  apiGroup: rbac.authorization.k8s.io
