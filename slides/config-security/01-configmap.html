Configuration

ConfigMap

  • A ConfigMap stores non‑sensitive configuration such as flags, environment variables, and app settings.
  • Never store secrets in ConfigMaps — they are not encrypted and readable by anyone with namespace access.
  • Use versioned ConfigMaps (e.g., api-config-v1, api-config-v2) to enable safe rollouts and rollbacks.
  • If your application does not reload config dynamically, you must restart Pods when the ConfigMap changes.
  • ConfigMaps can be consumed as environment variables, mounted files, or entire configuration directories.
  • Avoid large ConfigMaps — keep configuration small and focused to reduce rollout complexity.
  • Use immutable ConfigMaps for performance and safety when config does not change frequently.
# Versioned ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: api-config-v1
data:
  LOG_LEVEL: "info"
  FEATURE_FLAG: "true"

---

# Pod referencing a specific version
envFrom:
  - configMapRef:
      name: api-config-v1

---

# Updating config? Create a new version:
apiVersion: v1
kind: ConfigMap
metadata:
  name: api-config-v2
data:
  LOG_LEVEL: "debug"
  FEATURE_FLAG: "false"

# Then update your Deployment to reference:
#   api-config-v2
