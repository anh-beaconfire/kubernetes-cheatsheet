Configuration

Secret

  • A Secret stores sensitive values such as passwords, tokens, API keys, certificates, and private configuration.
  • Never commit raw Secrets to Git — use sealed secrets, SOPS, or an external secret manager.
  • Enable encryption at rest for Secrets using a KMS provider (AWS KMS, GCP KMS, Azure Key Vault, HashiCorp Vault).
  • Use external secret managers (Vault, AWS/GCP/Azure) for centralized, audited, auto‑rotated secrets.
  • Integrate external secrets using:
    • External Secrets Operator
    • Secrets Store CSI Driver
    • Vault Agent Injector
  • Rotate Secrets regularly — update the Secret and trigger a Deployment rollout.
  • Restrict access with RBAC and limit which Pods and users can read specific Secrets.
  • Avoid mounting Secrets as files if possible — environment variables are simpler but still readable by the process.
# Kubernetes Secret (opaque)
apiVersion: v1
kind: Secret
metadata:
  name: api-secret
type: Opaque
stringData:
  DATABASE_PASSWORD: "change-me"

---

# Pod consuming the Secret
env:
  - name: DATABASE_PASSWORD
    valueFrom:
      secretKeyRef:
        name: api-secret
        key: DATABASE_PASSWORD

---

# Example: External Secrets Operator (pulling from Vault)
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: api-db-password
spec:
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: api-secret
  data:
    - secretKey: DATABASE_PASSWORD
      remoteRef:
        key: kv/data/api
        property: db_password
